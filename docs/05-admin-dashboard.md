# Admin dashboard

The dashboard is feasible and implemented, but it is not needed for the initial offline release.
Deploy it when you need support or monetization operations.

## What “users” means

Most people using Spark have no server account and are invisible to the dashboard. A user appears
only after deliberately using support, purchase verification, promo checking, or another optional
cloud feature.

This is a privacy feature, not missing analytics. The dashboard cannot review:

- installed-device count
- habits or completions
- focus history
- routines
- capacity check-ins
- captured thoughts

Store statistics in Play Console provide install and purchase totals without copying behavior
data into Spark.

## Run locally

After completing cloud setup, create `apps/admin/.env.local` and run:

```powershell
npm.cmd run admin
```

Open [http://localhost:5173](http://localhost:5173). Its origin must appear in the Cloud Run
`ALLOWED_ORIGINS`.

## Dashboard sections

### Overview

Counts cloud identities, open support threads, and premium entitlements. These are bounded
aggregation queries triggered only on page load or Refresh.

### Support

An asynchronous private inbox:

- filter by status
- read messages
- reply
- resolve a thread

There is no typing indicator, online presence, or global real-time listener. This avoids the cost
and moderation needs of live chat while still feeling like an in-app conversation.

### Users

Shows the random Firebase UID, optional linked email, app/platform version, last cloud use, and
entitlement.

An owner may grant or revoke premium with a required audit reason. Support staff can assign an
official Play promo code already imported by an owner.

To grant a specific person:

1. Ask them to open Spark **Settings → Spark premium → Check staff or promo grant**, or to start a
   support conversation.
2. Locate the resulting UID or support thread.
3. Grant access with a reason.
4. Ask them to check access again.

An entirely offline installation cannot receive a remote grant because there is no identity to
grant.

### App config

Changes only bounded, non-sensitive global values:

- support enabled
- purchases enabled
- notification cap
- focus duration choices
- one global announcement

The mobile app caches config for 24 hours. Config must never contain per-user targeting or secret
values.

### Promo codes

Import codes generated by Google Play Console. The dashboard tracks available and assigned
inventory. Assigned codes remain Google Play entitlements; Spark does not issue a competing
public license.

### Admins

Owner-only role management:

- `support`: support and existing-code assignment
- `content`: configuration
- `owner`: grants, imports, roles, and all lower permissions
- `none`: remove access

Custom-claim changes take effect after the person signs out and in.

## Hosting

The dashboard compiles to static files:

```powershell
npm.cmd run build --workspace @spark/admin
npx.cmd firebase-tools deploy --only hosting
```

Firebase Hosting also publishes `privacy.html`. The dashboard's HTML is public, but all useful API
data remains behind Firebase authentication and server role checks.

## Operational routine

At small scale, check the dashboard manually:

- support: daily or on notification
- promo inventory: before campaigns
- config: only when a change is needed
- users/grants: on request
- Cloud Billing: weekly during testing, monthly after stable usage

Do not leave the dashboard open as a wallboard; it has no real-time updates anyway.
